= Міграція інстансу CICD на AWS EKS кластер.

== Скорочення
* Ci - Continuous Integration
* Cd - Continuous Delivery/Deployment
* OKD-CICD2 - версії 4.9, використовується для активної розробки продукту


== Передумови міграції
* Відмовитися від OKD 4.9, як основної платформи для розробки продукту, та перейти на AWS EKS.
* Перейти на більш економічну модель використання ресурсів AWS.
* Використання для розробки продукту версії EDP 3.x, замість існуючуї EDP 2.х
* Переход на Теkton Pipelines

== Процес міграції.

Для повної міграції усіх процессів налаштованих на OKD-CICD2 4.9 в AWS EKS,
процесс повинен бути розділенний на 2 частини і повинен бути налаштований один за одним.

Етапи міграції:

* Ci використовуючи AWS EKS та Cd використовуючи OKD-CICD2.
* Ci/Cd використовуючи AWS EKS та OKD-CICD2 як Cd оточення.

Розглянемо реалізацію двух етапів:

=== Ci використовуючи AWS EKS та Cd використовуючи OKD-CICD2.
Подана діаграма показує перший етап міграції і взаємодію між сервісамі в системі OKD-CICD2 та AWS EKS.

.Приклад інтеграції _EKS_ з _OKD-CICD2_ для Ci.
image::architecture/cicd-migration/cicd-migration.svg[]

Де:

* 1 - AWS EKS кластер розгорнутий Terraform.
* 2 - Ресурс Codebase та його похідні, відповідають за створенний бізнес застонусунок/бібіліотеку/інше.
Ресурс створенний з репозиторію Gerrit який знаходиться на кластері OKD-CICD2(4)
* 3 - Tekton пайплайни, для інтеграції з Gerrit та служать для проходження Code-Review та збірки(Build).
Згідно діаграми, Tekton налаштован на Nexus,Sonar,Docker Registry(5) в OKD-CICD2 кластері.
* 4 - OKD-CICD2 кластер версії 4.9 який використовується для Ci/Cd.
* 5 - Ci/Cd cервери такі які Gerrit,Nexus,Sonar розташовані в середовищі OKD-CICD2.
* 6 - Ресурси EDP, необхідні для процессу оновлення версій для бізнес компонент, і подальшому використання цих версій
в Cd процессі
* 7 - Сd процесс налаштованний на сервері Jenkins з використанням Groovy Shared Library.
* 8 - OKD кластер версії 4.12, використвується для розгортання платформи і тестування продукту.
* 9 - Механізм розгортання рееєстрів платформи за допомогою Control Plane Console і процесс тестування реєстрів.


На діаграмі (Figure 1)  можна побачити, що в першому етапі міграції, EDP розгорнутий в EKS оточенні, буде працювати з ресурсамви розташованими в
OKD-CICD2 оточенні і буде повністю інтегрований в Ci/Cd процесс в якому Сd виступає сервер Jenkins розгорнутий в OKD CICD2
що позволяє не змінювати підхід розгортання тестових оточень, збірки компонент, та збірки продукту(control-plane-installer)

Відносно кодової бази, усі зміни будуть йти в Gerrit (OKD CICD2) та при кожній збірці компонента буде налаштована реплікація в
Gerrit(EKS) що в подальшому позволить перейти на новий процесс розробки "без швів"

=== Ci/Cd використовуючи AWS EKS та OKD-CICD2 як Cd оточення
Подана діаграмма презентує фінальний етап міграції з OKD-CICD2 в оточення AWS EKS і використання оточення як основного
оточення для розробки.

.Приклад налаштування Ci/Cd в оточенні AWS EKS.
image::architecture/cicd-migration/cicd-eks-final-migration.svg[]

Де:

* 1 - Ресурс Codebase та його похідні, відповідають за створенний бізнес застонусунок/бібіліотеку/інше.
* 2 - Ci/Cd cервери такі які Gerrit,Nexus,Sonar розташовані в середовищі EKS де кодова база зберігається в Gerrit.
* 3 - Tekton пайплайни, для інтеграції з Gerrit та служать для проходження Code-Review та збірки(Build).
Згідно діаграми, Tekton налаштован на Nexus,Sonar,Docker Registry(2) в середині EKS.
* 4 - Tekton Pipelines відповідальні за розгортання OKD кластерів версії 4.12 та розгортання платформи
з архіва - *control-plane-installer*
* 5 - OKD кластери розгорнуті за допомогою Tekton(4). Кластери можуть оновлюватись мануально шляхом процедури оновлення платформи,
або оновлення за допомогою CD pipelines(6)
* 6 - Tekton CD pipelines відповідальні за створення/оновлення  ресурсів реєстрів/платформи.
* 7 - Tekton Pipeline відповідальний за збірку архіву control-plane-installer, після проходження sit пайплайну.
* 8 - Tekton Pipeline відповідальний за видалення кластерів після проведення тестування.


Якщо порівнювати процесси зі збіркою/розгортанням на AWS EKS з OKD CICD2, то в данному процессі для розгортання оточення, потрібно розгортати OKD кластер і працювати з реєстрам на кластері, як з CD оточенням.
Великий плюс данного підхода, це використання одного кластеру для тестування і динамічне оновлення компонент без потрібності перезбірки *control-plane-installera*, що в свою чергу призведе до економії часу на розробку,
та дозволить використовувати ресурси AWS без великого часу "простою".

Якщо казати про використання ресурсів кластером AWS EKS, то до нього застосовується модель використання Spot Instance що дозволить не використовувати ресурси AWS EC2.

Зі строни підтримки кластера розробки, буде застосован підхід GitOps, зі змогою оновлюватись до останніх версій компонент та тримати інфрастуктуру проекту в единій кодовій базі.\

== Використання ресурсів AWS

Для приблизного розуміння використання ресурсів AWS після міграції Ci/Cd.

.Приблизний прорахунок витрат на Сi/Cd з використанням AWS EKS.
image::architecture/cicd-migration/eks-cluster-costs-estimation.png[]

При прорахунку вартості, була врахована наявна інрафструктура і налаштування.
Для ресурсів EC2 було розраховано 2 моделі використання ресурсів - on-demand та spot.

* on-demand - модель застовується до інстансів з постійним доступом до них,  данна модель може бути застосована до EC2 інстансів які використовуються для розгортання на них ресурсів для постійного доступу до цих ресурсів (Gerrit,Nexus,Sonar,Moon e.t.c)
* spot - модель застосовується до інстансів які перебувають під постійним наватеженням і може бути застовоно для запуску Tekton Pipeline.

Дивлячись на данну модель використання AWS EKS для побудови процессу Ci/Cd, данна модель виглядає економічною за модель використання OKD-CICD2.
