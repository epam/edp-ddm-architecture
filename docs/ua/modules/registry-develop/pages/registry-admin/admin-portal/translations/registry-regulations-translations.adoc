== Загальний опис

Переклади елементів регламенту налаштовуються через інтерфейс *Кабінету адміністратора регламентів*, що дозволяє розробникам створювати локалізовані версії елементів бізнес-процесів, назв груп і елементів UI-форм відповідно до мови Кабінету користувача.

Для організації перекладів використовується бібліотека *`i18n`*, яка забезпечує обробку та завантаження перекладів для інтерфейсу користувача. У процесі налаштування перекладів розробник додає ключі та значення перекладів до відповідних JSON-файлів у директорії *_i18n_*, розташованій у регламенті реєстру. Ці файли можуть містити переклади назв компонентів, повідомлень та інтерфейсних елементів тощо, що дозволяє автоматично відображати локалізований контент відповідно до обраної мови в Кабінеті користувача.

Файли перекладів організовані за наступною структурою у регламенті реєстру. Це спрощує управління локалізацією та зберігає локалізовані значення для кожної мови у вигляді окремого JSON-файлу.

.Структура регламенту для управління перекладами
[plantuml]
----
@startsalt
{
{T
+ <&folder> **registry-regulations**
++ <&folder> **i18n**
+++ <&file> uk.json
+++ <&file> en.json
}
}
@endsalt
----

== Кроки налаштування перекладів

. Відкрийте menu:Кабінет адміністратора регламентів[Управління перекладами].
. Перейдіть на вкладку *Переклади регламенту*. Ви побачите перелік мов, доступних для перекладу.
+
NOTE: За замовчуванням, перелік містить файли для української та англійської мов, які після створення реєстру ще не містять жодних перекладів.
+
.Переклади регламенту
image::registry-admin/admin-portal/translations/registry-regulations-translations-1.png[]

. Додайте переклади для елементів послуг. +
Щоб забезпечити відображення елементів бізнес-процесів, груп та UI-форм відповідною мовою, додайте переклади для таких компонентів:

* Назви та елементи моделювання кожного бізнес-процесу, включно з текстами для валідаційних помилок, доданих через розширення xref:bp-modeling/bp/element-templates/service-task-templates/error-handling/throw-validation-error.adoc[Throw validation error].
* Назви, кнопки, підказки й тексти помилок кожної UI-форми.
* Назви груп бізнес-процесів.

+
[IMPORTANT]
====
Зверніть увагу, що передзаповнені значення (*default values*) та текст у компоненті xref:bp-modeling/forms/components/content.adoc[Content] не перекладаються автоматично при зміні мови у Кабінеті.
====

. Рекомендуємо додавати переклади під час моделювання.
При моделюванні бізнес-процесу та форм додавайте переклади у відповідні файли на вкладці *Переклади регламенту*. Це також стосується повної та скороченої назви реєстру.
+
TIP: Щоб уникнути окремих змін для кожного компонента, вносьте переклади у xref:registry-admin/admin-portal/version-control/candidate/create-new-change-request.adoc[версію-кандидат], а після завершення всіх змін -- застосовуйте до xref:registry-admin/admin-portal/version-control/master/master-version-settings.adoc[мастер-версії].

. Додайте новий переклад.
Для цього натисніть кнопку btn:[Додати переклад] навпроти потрібної мови.
+
.Додавання нового перекладу
image::registry-admin/admin-portal/translations/registry-regulations-translations-4.png[]
+
Для ключів перекладу компонентів використовуйте вихідні значення мовою, якою було змодельовано бізнес-процес. Для повної та скороченої назви реєстру -- значення, вказані в *Глобальних налаштуваннях* при створенні реєстру.
+
.Ключі для перекладу назв реєстру у глобальних налаштуваннях
image::registry-admin/admin-portal/translations/registry-regulations-translations-5.png[]

. Заповніть перекладені значення.
У вікні редагування введіть значення для кожного поля за допомогою бібліотеки *monaco-editor*, включаючи переклади для скороченої і повної назви реєстру.
+
.Редагування перекладів
image::registry-admin/admin-portal/translations/registry-regulations-translations-6.png[]

. Перевірка формату JSON.
Після введення даних відбувається валідація формату JSON. Кнопка btn:[Зберегти] стане доступною лише у випадку правильного формату.
+
.Валідація JSON
image::registry-admin/admin-portal/translations/registry-regulations-translations-7.png[]

== Використання базового перекладу для нової мови

Якщо переклад для певної мови вже створено, під час додавання перекладу для іншої мови можна обрати наявний переклад як базу для нового. Це дозволить розробнику зберегти ключі без змін і лише адаптувати значення до нової мови.

. Виберіть базову мову.
У полі *Переклад для якої мови взято за основу* показано мови, для яких уже створено переклади.
+
.Вибір базової мови
image::registry-admin/admin-portal/translations/registry-regulations-translations-13.png[]

. Змініть значення перекладу.
У вікні редагування відобразяться переклади з обраного файлу. Залиште ключі незмінними, замінивши лише значення для нової мови.

. Застосуйте переклади у версії-кандидат.
Щоб внести всі зміни до основної версії одночасно, застосуйте переклади до версії-кандидат, а потім перейдіть до *Огляду версії* та натисніть btn:[Застосувати зміни до мастер-версії].
+
TIP: Детальніше про застосування змін див. у розділі xref:registry-develop:registry-admin/admin-portal/version-control/candidate/overview-new-change-request.adoc#push-changes-master[Застосування змін до мастер-версії].

== Видалення перекладів

. Видаліть переклад.
Натисніть іконку видалення поруч із мовою, переклад для якої потрібно видалити.
+
.Видалення перекладу
image::registry-admin/admin-portal/translations/registry-regulations-translations-14.png[]

. Підтвердьте дію.
У новому вікні підтвердьте видалення, натиснувши btn:[Видалити].

. Застосуйте зміни до мастер-версії.
Якщо видалення було виконано у версії-кандидат, застосуйте зміни до основної версії у розділі *Огляд версії*, натиснувши btn:[Застосувати зміни до мастер-версії].
+
.Застосування змін
image::registry-admin/admin-portal/translations/registry-regulations-translations-15.png[]

== Оновлені переклади у Кабінетах користувачів

Після внесення змін та їх застосування, усі нові переклади або зміни відображатимуться у Кабінетах користувачів під час вибору відповідної локалі.
