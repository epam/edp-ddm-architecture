== Загальний опис

У цьому документі описані налаштування реєстрів для тестових середовищ або для умов низького навантаження, де основними користувачами є розробники. У таких випадках можна застосувати обмеження на ресурси, щоб зменшити їх використання.

Для розробників, які хочуть, щоб їх реєстр споживав мінімум ресурсів і розгортався на одній ноді, доступна *конфігурація за замовчуванням*. Нижче, у розділі xref:#config-global-registry[] наведено приклад сніпету для цієї конфігурації.

image:registry-admin/registry-min-config/cp-registry-min-default-config-1.png[]

Основні аспекти конфігурації:

* **Тільки для `dev`-режиму.** Ця опція доступна лише для розробницьких середовищ (`dev` mode) і не підтримується у production-режимі. Детальніше про режими розгортання можна дізнатися на сторінці xref:registry-admin/change-dev-prod-mode.adoc[].
* **Максимальна щільність за замовчуванням з версії Платформи `1.9.9`.** Усі нові реєстри розгортаються з максимальною щільністю, використовуючи інстанс із 1 нодою, що підтримує приблизно 60 сервісів. Кожен сервіс має основний контейнер і додатковий контейнер Istio Sidecar. Детальніше дивіться на сторінці xref:admin:control-plane/registry-management/control-plane-registry-resources.adoc[].
* **Мінімальна конфігурація до версії `1.9.8` включно вимагає щонайменше двох нод.** Кожна нода підтримує близько 30 сервісів. З версії `1.9.9` адміністратори можуть оновити реєстри до останньої версії, активуючи кнопку *`Конфігурація за замовчуванням`* у налаштуваннях реєстру в Control Plane.
* **Для старих реєстрів:** Адміністратори версій `1.9.8` та нижче можуть самостійно вирішувати, чи варто переходити на конфігурацію з максимальною щільністю перед оновленням до `1.9.9`.
* **Нові реєстри розгортатимуться з мінімальною конфігурацією ресурсів.** Після початкового розгортання адміністратори зможуть налаштувати збільшення ресурсів, якщо буде потреба.

NOTE: Ці налаштування актуальні для версій Платформи `1.9.9` та новіших.

== Налаштування ресурсів у файлі values.yaml

Більшість параметрів для компонентів реєстру можна налаштувати через файл `values.yaml`.

[#config-global-registry]
=== Конфігурація через секцію global.registry

Основні параметри налаштовуються у блоці _global.registry_ файлу `values.yaml`. Їх можна змінювати через інтерфейс Control Plane або безпосередньо в репозиторії Gerrit.

TIP: Детальніше про налаштування ресурсів через Control Plane див. на сторінці * xref:admin:control-plane/registry-management/control-plane-registry-resources.adoc[].

=== Налаштування для компонентів поза секцією global.registry

Для конфігурації `kafka-cluster` необхідно внести зміни безпосередньо у файл `deploy-templates/values.yaml`, додавши такі параметри:

[source, yaml]
----
kafka:
  jvmOptions:
    '-Xmx': 1500m
----

== Додаткові ручні налаштування

=== Налаштування concurrency та deploymentStrategy через helmfile

Для оптимізації розгортання реєстру через helmfile рекомендується налаштувати `concurrency` та `deploymentStrategy` у файлі `deploy-templates/values.yaml`, додавши такі параметри:

[source, yaml]
----
global:
  deploymentStrategy: Recreate
  helmfile:
    concurrency: 8
----

=== Nexus

Для збереження згенерованих артефактів реєстру доцільно використовувати платформний Nexus, що дозволяє оптимізувати використання ресурсів.

Це налаштування можна зробити у Control Plane через вкладку *Сховище артефактів*, обравши опцію *Платформне*, або безпосередньо вказавши у файлі `deploy-templates/values.yaml` значення _global.artifactsStorage_ як _platform_.

[source, yaml]
----
global:
  artifactsStorage: platform
----

NOTE: Опція платформного Nexus доступна з версії Платформи 1.9.9.

=== Restic

Для реєстрів із високою щільністю рекомендується зменшити стандартні налаштування Restic для зниження навантаження на ресурси.

NOTE: Ці зміни вплинуть на всі віртуальні машини кластера. Для реєстрів із великим обсягом даних слід перевірити процес резервного копіювання.

Для змін налаштувань Restic необхідно внести коміт у файлі `deploy-templates/values.gotmpl` у репозиторії _backup-management_.

[source, yaml]
----
restic:
  resources:
    limits:
      cpu: 1500m
      memory: 1Gi
    requests:
      cpu: 50m
      memory: 128Mi
----

Після коміту потрібно запустити пайплайн _cluster-mgmt_, що звільнить до 0.5 CPU для інших операцій реєстру.

=== Cluster Logging Collector

Налаштування FluentD Collectors для тестових середовищ можуть бути зменшені для оптимізації.

Зміни потрібно внести у файл `deploy-templates/logging-instance/templates/050-clo-instance.yaml` у репозиторії _logging_, додавши наступний блок:

[source, yaml]
----
collection:
  resources:
    limits:
      cpu: 500m
      memory: 512Mi
    requests:
      cpu: 50m
      memory: 128Mi
----

Після змін необхідно перезапустити пайплайн _cluster-mgmt_.

== Очікуваний результат

Для роботи реєстру з максимальною щільністю буде достатньо одного інстансу з параметрами *m5.2xlarge*.

NOTE: Під час тестування конфігурація реєстру використовувала до 6 CPU та до 28 GB пам'яті.

NOTE: У середовищі vSphere можливо налаштувати машину з ресурсами, ближчими до тестових значень (6 CPU/28 GB), ніж у *m5.2xlarge* (8 CPU/32 GB).

Детальний звіт щодо тестування продуктивності доступний тут: xref:testing:performance-testing/perf-report/1-9-10/perf-test-1-9-10-250-1.adoc[Переглянути звіт].

[next-steps-resources-optimization]
=== Подальші можливості оптимізації ресурсів

Можливий варіант запуску реєстру на машині *r5.xlarge* (4 CPU/32 GB), проте при цьому слід враховувати такі обмеження:

* Зниження продуктивності — час відгуку може досягати 5-6 секунд під навантаженням.
* Чутливість до навантаження, особливо при роботі понад 5 користувачів одночасно.
* Затримка підняття подів (~1 година).

== Наступні кроки

Після виконання всіх конфігурацій для остаточної перевірки реєстру виконайте наступне:

- Переконайтеся, що резервне копіювання працює належним чином після налаштування Restic.
- Перевірте процес оновлення реєстру. За необхідності додайте тимчасово ще один інстанс *m5.2xlarge* на час оновлення, який можна видалити після завершення процесу.
