= Конфігурація реєстру для розгортання з максимальною щільністю

include::platform:ROOT:partial$templates/document-attributes/default-set-ua.adoc[]

Для роботи реєстру з мінімальним навантаженням (наприклад, реєстри в розробці, де навантаження створюють лише розробники реєстру), можливим є використання більш жорстких лімітів на ресурси з метою зменшення ресурсів, необхідних для постійної роботи реєстру.

Дана стаття пропонує набір кроків з налаштування ресурсів для подібних випадків.

NOTE: Налаштування актуальні для версій Платформи 1.9.10 та пізніших

== Налаштування ресурсів реєстру через values.yaml реєстру

На рівні values.yaml реєстру можливо налаштувати необхідну конфігурацію для більшості компонентів реєстру.

=== Налаштування через секцію global.registry

Більшість налаштувань можливо виконати через блок _global.registry_ у values.yaml реєстру.

Налаштування можливо виконувати через UI Контрол Плейну або через центральний Геріт у репозиторії відповідного реєстру.

=== Налаштування ресурсів поза межами секції global.registry

Налаштування для компоненту _kafka-cluster_ можливо виконати лише через прямі зміни у файлі _deploy-templates/values.yaml_ реєстру.
Для цього необхідно додати блок

[source, yaml]
----
kafka:
  jvmOptions:
    '-Xmx': 1500m
----

== Додаткові ручні налаштування

=== Задання concurrency та deploymentStrategy для оптимізації розгортання реєстру за допомогою helmfile

Оптимальним варіантом є налаштування `concurrency` та `deploymentStrategy` через Геріт. Для цього необхідно у
_deploy-templates/values.yaml_
реєстру додати блок _global.helmfile.concurrency_ та _global.deploymentStrategy_, рекомендоване значення
`concurrency` - 8, а `deploymentStrategy` - Recreate .

[source, yaml]
----
global:
  deploymentStrategy: Recreate
  helmfile:
    concurrency: 8
----

=== Nexus

Платформа надає можливість обрати тип сховища, яке використовується для збереження згенерованих артефактів реєстру.

При розгортанні реєстру з максимальною щільністю важливо обрати використання платформного Nexus, оскільки це дасть можливість зберегти додаткові ресурси для реєстру.

Вказане налаштування можливо виконати на рівні Контрол Плейн у табі *Сховище артефактів*, де у полі *Тип сховища* необхідно обрати *Платформне*.

При налаштуванні напряму у файлі _deploy-templates/values.yaml_ реєстру необхідно змінити значення поля _global.artifactsStorage_ на _platform_.

[source, yaml]
----
global:
  artifactsStorage: platform
----

NOTE: Опція використання платформного Nexus замість реєстрового для економії ресурсів доступна з версії Платформи 1.9.9.

=== Restic

Поточна дефолтна конфігурація Restic є занадто великою для використання у реєстрах з максимальною щільністю. У зв'язку з цим її можливо зменшити

NOTE: Вказані зміни застосуються на всі віртуальні машини кластеру і вплинуть на всі реєстри. Для реєстрів що зберігають велику кількість даних бажано перевірити процес створення бекапу з новими значеннями ресурсів Restic.

У репозиторії _backup-management_ центрального Геріта необхідно виконати коміт у _deploy-templates/values.gotmpl_ зі змінами блоку restic

[source, yaml]
----
  restic:
    resources:
      limits:
        cpu: 1500m
        memory: 1Gi
      requests:
        cpu: 50m
        memory: 128Mi
----

Після коміту необхідно запустити пайплайн _cluster-mgmt_.

Даний крок є опційним, але бажаним, оскільки звільняє до 0.5 CPU для інших операцій з реєстром.

=== Cluster Logging Collector

Поточна конфігурація FluentD Collectors є неефективною з точки зору розробницького інстансу Платформи та її можна зменшити.

У репозиторії _logging_ центрального Геріта необхідно виконати коміт у _deploy-templates/logging-instance/templates/050-clo-instance.yaml_ зі змінами блоку collection:
[source, yaml]
----
  collection:
    resources:
      limits:
        cpu: 500m
        memory: 512Mi
      requests:
        cpu: 50m
        memory: 128Mi
----

та перезапустити пайплайн _cluster-mgmt_.

== Планована вигода

Для роботи реєстру, конфігурованого з максимальною щільністю, має бути достатнім *1* інстанс машини розміром *m5.2xlarge*.

NOTE: При тестуваннях з вказаною конфігурацією реєстр використовував до 6 CPU та до 28 GB пам'яті

NOTE: При роботі з Платформою у vSphere можливо налаштувати машину з кількістю ресурсів, що відрізняються від m5.2xlarge (8 CPU/32 GB) та є ближчими до значень, отриманих при тестуванні (6 CPU/28 GB).

Тестування продуктивності, проведене на вказаних налаштуваннях: xref:testing:performance-testing/perf-report/1-9-10/perf-test-1-9-10-250-1.adoc[репорт]

== Наступні кроки

Після виконання усіх конфігурацій, описаних вище, для остаточної перевірки працездатності реєстру, необхідно виконати наступні кроки:

- Перевірити бекапування реєстрів на кластері після конфігурації _Restic_
- Перевірити кейс оновлення реєстру (за необхідності, на час оновлення реєстру можливо додавати ще 1 машину m5.2xlarge, яку можна видалити після оновлення)
