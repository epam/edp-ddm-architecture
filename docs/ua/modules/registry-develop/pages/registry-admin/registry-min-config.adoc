Для роботи реєстру з мінімальним навантаженням (наприклад, реєстри в розробці, де навантаження створюють лише розробники реєстру), можливим є використання більш жорстких лімітів на ресурси з метою зменшення ресурсів, необхідних для постійної роботи реєстру.

== Налаштування ресурсів через values.yaml реєстру

На рівні values.yaml реєстру можливо налаштувати необхідну конфігурацію для більшості компонентів реєстру.

== Додаткові ручні налаштування

=== Kafka-cluster

NOTE: Автоматизація даного кроку планується в майбутніх релізах

Налаштування обмеження ресурсів для Кафки є дуже важливим, оскільки вона є одним із найбільших споживачів ресурсів та без обмежень може їх споживати надлишково.

У values.yaml реєстру наразі можливо налаштувати ліміти лише для власне _kafka-cluster_ контейнера (буде виправлено в майбутніх релізах).
Проте без відповідного налаштування jvm опцій контейнера досягти коректної роботи реєстру з максимальною щільністю буде досить складно.

Для налаштування необхідних параметрів необхідно зробити коміт у центральний Геріт у репозиторій _strimzi-kafka-operator_ файл _deploy-templates/templates/kafka-cluster.yaml_

[source, yaml]
----
spec:
  kafka:
    jvmOptions:
      '-Xmx': 1500m
----

Якщо конфігурація з максимальною щільністю планується до розповсюдження на усіх реєстрах кластера, то коміт можливо виконувати у релізну гілку компоненту. Інакше необхідне створення окремої гілки з останнього релізу і її вказання у helmfile реєстру.

Після коміту змін необхідно запустити пайплайн деплою реєстру.

=== Nexus

Ситуація для Nexus є схожою на Kafka.

Поточний механізм налаштування ресурсів не дає можливості керувати опціями jvm, які є критичними для коректного управління ресурсами Java-застосунків.
Проте Nexus використовується при роботі реєстру не настільки активно, тому кількість ресурсів, яку він споживає, не є настільки критичною для роботи реєстру з максимальною щільністю.

Відповідно, даний крок з налаштування ресурсів є опційним

==== Створення конфігмапи з jvm опціями

В репозиторії _nexus_ у центральному Геріті необхідно додати файл _deploy-templates/templates/cm/configuration/nexus_vmoptions.yaml_

====
----

----
====

Після цього необхідно примаунтити дану конфігмапу до деплоймента Nexus. У файлі _deploy-templates/templates/nexus-deployment.yaml_ додати у spec.template.spec.containers.nexus.volumeMounts

[source, yaml]
----
            - name: nexus-vm-options
              mountPath: /opt/sonatype/nexus/bin/nexus.vmoptions
              subPath: nexus.vmoptions
----

та у spec.template.spec.volumes

[source, yaml]
----
        - name: nexus-vm-options
          configMap:
            name: nexus-vm-options
            defaultMode: 420
----

Якщо конфігурація з максимальною щільністю планується до розповсюдження на усіх реєстрах кластера, то коміт можливо виконувати у релізну гілку компоненту. Інакше необхідне створення окремої гілки з останнього релізу і її вказання у helmfile реєстру.

Після коміту змін необхідно запустити пайплайн деплою реєстру.

=== Restic

Поточна дефолтна конфігурація Restic є занадто великою для використання у реєстрах з максимальною щільністю. У зв'язку з цим її можливо зменшити

NOTE: Вказані зміни вплинуть на бекапування усіх реєстрів кластеру, працездатність бекапування бажано перевірити після застосування конфігурації

У репозиторії _backup-management_ центрального Геріта необхідно виконати коміт у _deploy-templates/values.gotmpl_ зі змінами блоку restic

[source, yaml]
----
  restic:
    resources:
      limits:
        cpu: 1500m
        memory: 1Gi
      requests:
        cpu: 50m
        memory: 128Mi
----

Після коміту необхідно запустити пайплайн _cluster-mgmt_.

Даний крок є опційним, але бажаним, оскільки звільняє до 0.5 CPU для інших операцій з реєстром.

== Планована вигода

Для роботи реєстру, конфігурованого з максимальною щільністю, має бути достатнім *1* інстанс машини розміром *r5.2xlarge*

== Наступні кроки

Після виконання усіх конфігурацій, описаних вище, для остаточної перевірки працездатності реєстру, необхідно виконати наступні кроки:

- Перевірити бекапування реєстрів на кластері після конфігурації _Restic_
- Перевірити кейс оновлення реєстру (рекомендація: для більш швидкого та оптимального оновлення реєстру може знадобитися ще 1 машина r5.2xlarge, яку можна видалити після оновлення)
