Як адміністратор реєстру, ви можете налаштувати зовнішню автентифікацію для Кабінету надавача та отримувача послуг, використовуючи зовнішнього постачальника ідентифікації (IDP) через протокол OpenID Connect. Посібник надає покрокові інструкції для налаштування цієї функції.

== Доступ до налаштувань Кабінетів надавача та отримувача послуг

NOTE: Для прикладу, розглянемо процес налаштування для *Кабінету надавача послуг*.

. Відкрийте
include::platform:ROOT:partial$templates/links/platform/administrative/control-plane.adoc[]
та перейдіть до режиму редагування реєстру.

. Виберіть вкладку *Кабінет надавача послуг* або *Кабінет отримувача послуг*, залежно від необхідності.

== Вкажіть тип автентифікації

NOTE: Налаштування альтернативного методу автентифікації здійснюється в адмін-панелі Control Plane окремо для кожного порталу.

. Знайдіть поле з назвою *Тип автентифікації*.
. Відкрийте список і виберіть один із наступних варіантів:

* **Базова автентифікація** (_за замовчуванням_) — Користувачі автентифікуються відповідно до поточної реалізації.
* **Зовнішній IDP (OIDC)** — Користувачі автентифікуються через зовнішнього постачальника ідентифікації, використовуючи протокол OpenID Connect.

+
image:registry-admin/cp-auth-setup-officers/specify-auth-type/op-specify-auth-type-global-1-ua.png[]

+
[NOTE]
====
За замовчуванням вибрано **Базову автентифікацію**, що означає, що автентифікація користувачів відбувається згідно з поточною реалізацією.

image:registry-admin/cp-auth-setup-officers/specify-auth-type/op-specify-auth-type-global-2.png[]
====

== Налаштуйте параметри зовнішнього постачальника ідентифікації (IDP)

Якщо ви вибрали *Зовнішній IDP*, виконайте наступні кроки для налаштування необхідних параметрів.

=== Передумови

Переконайтеся, що виконані наступні умови перед продовженням:

* Конфігурація клієнта повинна бути попередньо налаштована у зовнішнього постачальника ідентифікації.
* Взаємодія між зовнішнім і внутрішнім постачальниками повинна бути належним чином налаштована.

=== Обов'язкові поля

Заповніть наступні обов'язкові поля інформацією з конфігурації клієнта зовнішнього постачальника:

|===
|Параметр/Поле |Опис |Приклад

|*Authorization URL*
|Authorization URL — це кінцева точка зовнішнього постачальника ідентифікації, де починається процес автентифікації. Це URL, на який користувачі перенаправляються для входу. Зазвичай цей URL веде на сторінку входу, де користувачі вводять свої облікові дані. Постачальник ідентифікації використовує цей URL для ініціювання процесу отримання дозволу від користувача.
|`https://idp.example.com/auth`

|*Token URL* (_створеного клієнта_)
|Token URL — це кінцева точка зовнішнього постачальника ідентифікації, де код авторизації обмінюється на токен доступу. Цей токен доступу буде використовуватися вашим додатком для автентифікації подальших запитів до сервера ресурсів.
|`https://idp.example.com/token`

|*Client ID*
|Client ID — це унікальний ідентифікатор, призначений вашому додатку зовнішнім постачальником ідентифікації. Він використовується для ідентифікації додатка, що здійснює запит на автентифікацію.
|`my-application-client-id`

|*Client Secret*
|Client Secret — це конфіденційний ключ, яким обмінюються ваш додаток і зовнішній постачальник ідентифікації для підтвердження автентичності запиту.
|`my-client-secret`
|===

|*Issuer*
|Issuer — це унікальний ідентифікатор зовнішнього постачальника ідентифікації. Зазвичай це URL служби постачальника.

|Logout URL — це кінцева точка для виходу з системи, що використовується для завершення сесії у постачальника ідентифікації.

|Перемикач дозволяє чи забороняє функцію *Back-channel Logout*, яка забезпечує завершення сесії на всіх платформах.

== Підтвердьте та застосуйте зміни

. Після заповнення всіх обов'язкових полів натисніть кнопку *`Підтвердити`*.
+
image:registry-admin/cp-auth-setup-officers/specify-auth-type/op-specify-auth-type-global-4-ua.png[]

. Перейдіть до розділу *Запити на зміни* і підтвердьте зміни.
. Дочекайтеся успішного завершення збірки. Компоненти реєстру будуть оновлені, і Кабінет користувача почне використовувати зовнішнього постачальника для автентифікації.
+
image:registry-admin/cp-auth-setup-officers/specify-auth-type/op-specify-auth-type-global-5.png[]

== Важливі примітки

* Користувачі, створені за допомогою *Basic Auth*, не зможуть автентифікуватися в порталі після переходу на *External IDP*.
* Після автентифікації через зовнішнього постачальника ідентифікації, в реєстрі автоматично створюється новий користувач у відповідному Keycloak-реалмі з фіктивними службовими атрибутами `fullName`, `drfo` та `edrpou` і роллю за замовчуванням -- `citizen` або `officer`.
* *Управління користувачами* виконується на рівні системи, яка виконує роль зовнішнього IDP.
* *Управління ролями користувачів*, які автентифікувались через зовнішній IDP, виконується на рівні відповідних реалмів реєстру.
