== Загальний опис

*_Knative функції_* -- це можливість створювати *Serverless*-застосунки для кластерів на основі Kubernetes. Вони є open-source-аналогом для таких популярних рішень як AWS Lambda, Google Cloud Functions та інших.

У регламент реєстру можна додавати функції, які працюють на основі Knative Serving, тобто обробляють HTTP-запити. Ці функції можуть бути написані на різних мовах програмування/фреймворках, зокрема:

* Javascript
* Typescript
* C#
* Go
* Java (Spark)
* Java (Spring)
* Kotlin
* PHP
* Python
* Ruby
* Scala
* Shell

[TIP]
  Детальніше ознайомитись з Knative Serving можна на https://knative.dev/docs/serving/[офіційному сайті].

== Увімкнення Knative-функцій Платформі

Якщо ви хочете використовувати Knative-функції у вашому реєстрі, необхідно увімкнути таку можливість на рівні налаштувань Платформи. Це можна зробити наступним чином:

. Відкрийте адміністративний інтерфейс
include::platform:ROOT:partial$templates/links/platform/administrative/control-plane.adoc[]
.

. Перейдіть до розділу *Керування Платформою* та натисніть *`Редагувати`* у правому верхньому куті.
. Далі відкрийте секцію menu:Налаштування Платформи[Serverless] та активуйте перемикач в положення `ON`.
+
NOTE: Перед вимкненням опції Serverless, переконайтеся, що жоден з реєстрів не використовує Knative-функції.
+
image:bp-modeling/bp/knative/knative-02.png[]

. Натисніть *`Підтвердити`*, щоб зберегти зміни. В результаті система сформує новий запит на оновлення.
. Поверніться до розділу menu:Керування Платформою[Запити на оновлення], відкрийте відповідний запит та підтвердьте зміни до налаштувань.
+
Налаштування Serverless-оператора будуть додані у файл *_deploy-templates/values.yaml_* компонента `cluster-mgmt`:
+
[source,yaml]
----

  enabled: true
----
+
Зміни у запиті на оновлення виглядатимуть так:
+
image:bp-modeling/bp/knative/knative-03.png[]

. В результаті підтвердження, система запустить пайплайн *Master-Build-cluster-mgmt*, який виконає збірку коду і додасть відповідні зміни до налаштувань Платформи. Дочекайтеся виконання збірки. Це може зайняти кілька хвилин.

== Налаштування оточення для роботи

Для локальної розробки та тестування роботи функцій на кластері, потрібно встановити *Docker*, *Kubernetes CLI*, *OpenShift CLI*, *Knative CLI*, та *NodeJS*.

Зверніться до офіційної документації відповідних інструментів, щоб отримати детальні інструкції щодо їх встановлення, адже цей процес сильно залежить від вашої ОС (Windows, MacOS, Linux тощо).

[TIP]
Поточний документ концентрується на описі роботи із функціями, які написані мовою  Javascript (Typescript). При виборі іншої мови програмування для написання Knative-функцій, NodeJS встановлювати не потрібно, натомість необхідно встановити інструменти для роботи з обраною мовою програмування.

[#create-knative-function]
== Створення Knative-функції

Після коректної інсталяції Knative CLI, у вашому терміналі з'явиться доступ до утиліти `kn func` або просто `func`. За допомогою цієї утиліти можна створювати, видаляти, збирати та розгортати функції.

🛠️ *Створіть свою першу Knative-функцію за допомогою команди:*

[source,bash]
----
kn func create -l typescript my-first-knative-func
----

Команда `kn func create -l typescript my-first-knative-func` дозволяє створити нову функцію у середовищі Knative, використовуючи *Typescript* як мову програмування. Це початковий крок для розробки serverless-додатків, який автоматизує багато аспектів налаштування та розгортання.

При виконанні цієї команди відбувається кілька дій:

* Створюється нова директорія з назвою `my-first-knative-func`, яка слугуватиме контейнером для всього необхідного для вашої функції.
* У цій директорії автоматично генеруються початкові файли проєкту, включаючи вихідний код на Typescript, конфігураційні файли для Knative та документацію у форматі README, що допомагає новим користувачам розібратися з основами роботи з проєктом. +
`my-first-knative-func` -- це не лише назва директорії, яка буде створена в вашій файловій системі, але й ідентифікатор вашої функції у Knative екосистемі.
* Вказівка `-l typescript` означає, що для розробки функції обрана мова програмування Typescript. Це дозволяє використовувати типізацію у вашому коді, що може полегшити розробку та знизити кількість помилок у майбутньому. Змініть `typescript` на `node`, якщо вам зручніше працювати з JavaScript.

NOTE: Якщо потрібно змінити назву функції, ви можете зробити це пізніше, у конфігураційному файлі *_func.yaml_*. Вибір назви функції важливий, бо назва буде присутня в URL, за яким буде доступна функція.

Після виконання цієї команди, утиліта `func` створить готовий до запуску та розгортання проєкт, згідно зі вбудованим в *Knative CLI* шаблоном. Якщо вам потрібно більше опцій або ви бажаєте використати власний шаблон, ознайомтеся з документацією `kn func create`, виконавши у терміналі команду із параметром виклику довідки `--help`:

[source,bash]
----
kn func create --help
----

=== Локальний запуск

Підійде для перевірки та запуску функцій, які не потребують ресурсів кластера/Платформи/реєстру. Цей спосіб також буде зручним для виконання `debug` процедур, адже в цьому випадку функції виконуються повністю локально, і залучається мінімум інструментів, які також можуть викликати помилки. Такий режим запуску аналогічний запуску звичайного NodeJS-застосунку. Цей спосіб також підтримує 2 режими запуску:

Javascript Only::
Тут мається на увазі, що функція при запуску залучає лише Javascript-інструменти та бібліотеки. Для запуску в такому режимі, послідовно виконайте 3 команди у терміналі:

* Встановлення модулів *JS*, потрібно запустити тільки перший раз:
+
[source,bash]
----
npm i
----

* Збірка проєкту. Потрібна зажди після змін у коді, але взагалі не потрібна, якщо ваша функція не використовує Typescript.
+
[source,bash]
----
npm run build
----

* Запуск функції.
+
[source,bash]
----
npm run start
----

Run in docker::
У цьому випадку NodeJS код пакується в Docker контейнер ідентичний тому який буде працювати на кластері, але запускається локально. Для запуску в такому режимі виконайте наступну команду в терміналі:
+
[source,bash]
----
kn func run
----
+
Ця функція виконає збірку docker image, встановить необхідні модулі *NodeJS*, та запустить вашу функцію в Docker-контейнері.
+
При запуску в такому режимі, *Knative CLI* запитає адресу вашого *docker registry*. Якщо ви не збираєтеся запускати цю функцію на кластері, а просто хочете запустити та перевірити роботу функції, ви можете вказати будь-яку адресу, наприклад, `example.com/my-apps`.
+
Щоб кожного разу при запуску не вписувати адресу *docker registry*, скористайтеся опцією `--registry` або `-r`. В такому разі команда запуску виглядатиме так:
+
[source,bash]
----
kn func run -r example.com/my-apps
----
+
Після запуску в терміналі ви побачите лог роботи функції та *HTTP*-адресу за якою ви можете викликати функцію. За замовчуванням це буде `+http://localhost:8080+`. Ви також можете використати ваш браузер або *Postman* (чи аналогічне ПЗ) для тестування вашої функції.
+
NOTE: Якщо ви не збираєтеся тестувати вашу функцію, запускаючи її на кластері, то *Локального запуску* вам буде достатньо.
+
На цьому етапі ваша функція повністю готова для додавання в регламент реєстру (цей процес описано нижче). Якщо вам потрібно взаємодіяти із Вашої функції зі змінними бізнес-процесу або з API сервісів реєстру, то для тестування її роботи вам знадобиться підхід із *Запуском на кластері*.

==== Передумови

Для постійної та правильної роботи Knative-функції, її потрібно додати у регламент вашого реєстру. Проте для спрощення тестування роботи функції, ви можете тимчасово додати її на ваш кластер. Для цього вам знадобиться мати доступ до *OpenShift*.

[WARNING]
====
Станом на зараз для виконання розгортання функції через Knative CLI, а не шляхом додавання функції у регламент, необхідно розширити правило для ролі _Адміністратора регламенту_.

  - verbs:
      - get
      - list
      - watch
----

*на:*

[, yaml]
----
rules:
  - verbs:
      - '*'
----

== Додавання конфіг-мап та секретів Openshift в Knative функцію
Після створення функції, в директорії функції з'являється конфігураційний файл функції `func.yaml`. Він дозволяє нам налаштувати зокрема використання секретів та конфіг-мап *Openshift*. Розглянемо приклад додавання секретів в env змінні функції
[, yaml]
----
run:
  envs:
  - name: REDIS_PASSWORD
    value: '{{ secret:redis-auth:password }}'
----

В полі `value`, ми бачимо інструкцію для Knative, яка бере значення секрету та зберігає його в env змінну `REDIS_PASSWORD`

* `secret` - ключове слово, яке вказує на те що буде використано *OpenShift* секрет, замість `secret` тут може бути вказано `configMap`, що буде означати *OpenShift* конфіг-мапу.
* `redis-auth` - назва секрету
* `password` - поле в секреті, яке буде використано

В результаті цих налаштувань, в вашому коді Javascript функції, секрет буде доступний в змінній `process.env.REDIS_PASSWORD`

[TIP]
====
Для детального ознайомлення з налаштуваннями в файлі `func.yaml` зверніться до офіційної документації https://github.com/knative/func/blob/main/docs/reference/func_yaml.md#local-environment-variables[за посиланням]
====

== Приклади використання Knative-функцій

Для демонстрації роботи Knative функцій було написано 2 референтних бізнес-процеси. Вони написані з використанням `typescript` та працюють з дата-фабрикою

* *reference-knative-read-transform* - простий приклад отримання даних з дата-фабрики, з перетворенням масиву отриманих даних
* *reference-knative-write-transform* - приклад зміни даних колонки у всій таблиці дата-фабрики. Читання та запис в дата-фабрику, інтеграція з *Redis* та *digital-signature-ops* сервісами

== Розгортання Knative-функцій у реєстрі

Для використання створеної Вами функції в реєстрі, вона має бути доставлена в Ваш регламент реєстру. Функція являє собою директорію, в якій знаходиться вихідний код функції та її конфігураційні файли. Цю директорію потрібно додати в git-репозиторій вашого регламенту. Функції додаються в директорію *_knative_*. Якщо такої директорії ще не існує, її необхідно створити.

.Knative-функції у регламенті реєстру
image::bp-modeling/bp/knative/knative-01.png[]

[TIP]
====
Якщо Ви робили локальний запуск функції, то в директорії вашої функції буде створена директорія *node_modules* (в ній зберігаються javascript модулі необхідні  для локального запуску), яку варто видалити. Вона займає багато місця на диску, та не використовується при розгортанні в реєстрі

Також варто перевірити вміст вашого файлу `func.yaml`. При локальному запуску функції там можуть створитися зайві ключі які можуть впливати на розгортання функції. Варто видалити такі ключі `registry`, `image`, `imageDigest`, та `deploy`.
