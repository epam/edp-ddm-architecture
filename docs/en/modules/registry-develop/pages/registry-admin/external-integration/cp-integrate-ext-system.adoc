= Configuring integrations with external systems in Control Plane

[WARNING]
====
//–ù–∞–ª–∞—à—Ç—É–≤–∞–Ω–Ω—è –≤–∑–∞—î–º–æ–¥—ñ—ó —á–µ—Ä–µ–∑ –∞–¥–º—ñ–Ω—ñ—Å—Ç—Ä–∞—Ç–∏–≤–Ω—É –ø–∞–Ω–µ–ª—å Control Plane –¥–æ—Å—Ç—É–ø–Ω–µ –¥–ª—è –≤–µ—Ä—Å—ñ–π —Ä–µ—î—Å—Ç—Ä—É 1.9.3 —ñ –≤–∏—â–µ.
Configuring integrations through the Control Plane admin console is available for registries version 1.9.3 or later.

//–î–ª—è –≤–µ—Ä—Å—ñ–π —Ä–µ—î—Å—Ç—Ä—É 1.9.2 —Ç–∞ –Ω–∏–∂—á–µ –Ω–∞–ª–∞—à—Ç—É–≤–∞—Ç–∏ —ñ–Ω—Ç–µ–≥—Ä–∞—Ü—ñ—é –º–æ–∂–ª–∏–≤–æ –Ω–∞ —Ä—ñ–≤–Ω—ñ —Ä–µ–≥–ª–∞–º–µ–Ω—Ç—É —Ä–µ—î—Å—Ç—Ä—É —É —Ñ–∞–π–ª—ñ _bp-trembita/configuration.yml_ (_–∑–≤–µ—Ä–Ω—ñ—Ç—å—Å—è –¥–æ —Å—Ç–æ—Ä—ñ–Ω–∫–∏ xref:registry-develop:bp-modeling/bp/rest-connector.adoc#regulations-configuration[REST-–∫–æ–Ω–µ–∫—Ç–æ—Ä: –Ω–∞–ª–∞—à—Ç—É–≤–∞–Ω–Ω—è —Ä–µ–≥–ª–∞–º–µ–Ω—Ç—É]_).
For registries version 1.9.2 or earlier, you can configure integrations at the registry regulations level using the _bp-trembita/configuration.yml_ file. For details, see xref:registry-develop:bp-modeling/bp/rest-connector.adoc#regulations-configuration[REST connector: configuring the regulations].
====

//== –ó–∞–≥–∞–ª—å–Ω–∏–π –æ–ø–∏—Å
== Overview

//–©–æ–± –º–∞—Ç–∏ –∑–º–æ–≥—É –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É–≤–∞—Ç–∏ —Ä–æ–∑—Ä–æ–±–ª–µ–Ω–∏–π –Ω–∞ –ü–ª–∞—Ç—Ñ–æ—Ä–º—ñ —ñ–Ω—Ç–µ–≥—Ä–∞—Ü—ñ–π–Ω–∏–π REST-–∫–æ–Ω–µ–∫—Ç–æ—Ä –¥–æ –∑–æ–≤–Ω—ñ—à–Ω—ñ—Ö —Å–µ—Ä–≤—ñ—Å—ñ–≤ —Ç–∞ –æ—Ç—Ä–∏–º—É–≤–∞—Ç–∏ —ñ–Ω—Ñ–æ—Ä–º–∞—Ü—ñ—é –≤—ñ–¥ –∑–æ–≤–Ω—ñ—à–Ω—ñ—Ö —Å–∏—Å—Ç–µ–º —Ç–∞ —Ä–µ—î—Å—Ç—Ä—ñ–≤, –Ω–µ–æ–±—Ö—ñ–¥–Ω–æ –ø–æ–ø–µ—Ä–µ–¥–Ω—å–æ –≤–∏–∫–æ–Ω–∞—Ç–∏ –∫–æ–Ω—Ñ—ñ–≥—É—Ä–∞—Ü—ñ—ó –Ω–∞ —Ä—ñ–≤–Ω—ñ –µ–∫–∑–µ–º–ø–ª—è—Ä–∞ —Ä–µ—î—Å—Ç—Ä—É.
To use the Platform's REST connector for integration with external systems and registries and fetch data from them, you must first configure the connector at the registry instance level.

//–î–ª—è —Ç–æ–≥–æ, —â–æ–± –Ω–∞–ª–∞—à—Ç—É–≤–∞—Ç–∏ –≤–∑–∞—î–º–æ–¥—ñ—é —ñ–∑ –∑–æ–≤–Ω—ñ—à–Ω—ñ–º–∏ —Ä–µ—î—Å—Ç—Ä–∞–º–∏ —Ç–∞ —Å–∏—Å—Ç–µ–º–∞–º–∏, –Ω–µ–æ–±—Ö—ñ–¥–Ω–æ —Å—Ç–≤–æ—Ä–∏—Ç–∏ –∫–æ–Ω—Ñ—ñ–≥—É—Ä–∞—Ü—ñ—é —É –∫–æ–Ω—Å–æ–ª—ñ Control Plane. –¶–µ –¥–æ–∑–≤–æ–ª–∏—Ç—å —Ä–æ–∑–≥–æ—Ä–Ω—É—Ç–∏ *`Service Entry`* —Ç–∞ –≤—ñ–¥–∫—Ä–∏—Ç–∏ –¥–æ—Å—Ç—É–ø –¥–ª—è —Ç—Ä–∞—Ñ—ñ–∫—É –∑–∞ –º–µ–∂—ñ –∫–ª–∞—Å—Ç–µ—Ä–∞ –ü–ª–∞—Ç—Ñ–æ—Ä–º–∏, –∞ —Ç–∞–∫–æ–∂ —Å—Ç–≤–æ—Ä–∏—Ç–∏ —Å–µ–∫—Ä–µ—Ç –¥–ª—è –¥–æ—Å—Ç—É–ø—É –¥–æ —Ä–µ—Å—É—Ä—Å—ñ–≤ –∑–æ–≤–Ω—ñ—à–Ω—å–æ—ó —Å–∏—Å—Ç–µ–º–∏.
To configure external interactions, you need to create a corresponding configuration in the Control Plane console. This enables deploying a *Service Entry*, which serves as an entry point for the outbound traffic, and creating a secret for accessing the external resources.

[TIP]
====
//* Service Entry —Å—Ç–≤–æ—Ä—é—î—Ç—å—Å—è –∞–≤—Ç–æ–º–∞—Ç–∏—á–Ω–æ, –ø—Ä–∏ —Ä–æ–∑–≥–æ—Ä—Ç–∞–Ω–Ω—ñ —Ä–µ—î—Å—Ç—Ä—É –∑–∞ –¥–æ–ø–æ–º–æ–≥–æ—é Jenkins-–ø–∞–π–ø–ª–∞–π–Ω—É -- `*MASTER-Build-<registry-name>*` (_–¥–µ—Ç–∞–ª—å–Ω—ñ—à–µ -- –¥–∏–≤. —Ä–æ–∑–¥—ñ–ª xref:registry-develop:bp-modeling/bp/rest-connector.adoc#create-service-entry[–°—Ç–≤–æ—Ä–µ–Ω–Ω—è ServiceEntry]_).
* A Service Entry is created automatically when a registry is deployed using the *MASTER-Build-`<registry-name>`* Jenkins pipeline. For details, see xref:registry-develop:bp-modeling/bp/rest-connector.adoc#create-service-entry[Rest connector: creating a Service Entry].
//* –°–µ–∫—Ä–µ—Ç–∏ (—Ç–æ–∫–µ–Ω, –ø–∞—Ä–æ–ª—å —Ç–æ—â–æ) –∞–Ω–∞–ª–æ–≥—ñ—á–Ω–æ —Å—Ç–≤–æ—Ä—é—é—Ç—å—Å—è –∞–≤—Ç–æ–º–∞—Ç–∏—á–Ω–æ –ø—ñ—Å–ª—è –∑–∞—Å—Ç–æ—Å—É–≤–∞–Ω–Ω—è –Ω–∞–ª–∞—à—Ç—É–≤–∞–Ω—å –ø–∞–π–ø–ª–∞–π–Ω–æ–º Jenkins -- `*MASTER-Build-<registry-name>*`. –í—ñ–Ω –¥–æ–¥–∞—î—Ç—å—Å—è –¥–æ *user-management:hashicorp-vault* –¥–ª—è —Ç—ñ—î—ó —Å–∏—Å—Ç–µ–º–∏/—Å–µ—Ä–≤—ñ—Å—É, –¥–æ —è–∫–æ—ó –Ω–µ–æ–±—Ö—ñ–¥–Ω–æ –≤–∏–∫–æ–Ω—É–≤–∞—Ç–∏ –∑–∞–ø–∏—Ç–∏ (_–¥–µ—Ç–∞–ª—å–Ω—ñ—à–µ -- –¥–∏–≤. —Ä–æ–∑–¥—ñ–ª xref:registry-develop:bp-modeling/bp/rest-connector.adoc#create-secret[–°—Ç–≤–æ—Ä–µ–Ω–Ω—è —Å–µ–∫—Ä–µ—Ç—É –¥–ª—è –∞–≤—Ç–æ—Ä–∏–∑–∞—Ü—ñ—ó —Å–µ—Ä–≤—ñ—Å—É]_).
* Similarly, the secrets (such as a token or password) are created automatically after the settings are applied using the *MASTER-Build-`<registry-name>`* Jenkins pipeline. The secret is added to the *user-management:hashicorp-vault* for the system or service to which requests will be made. For details, see xref:registry-develop:bp-modeling/bp/rest-connector.adoc#create-secret[Rest connector: creating a secret for service authorization].
====

`NO_AUTH`::
//–ú–µ—Ç–æ–¥/—Ç–∏–ø –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —è–∫–∏–π –Ω–µ –ø–æ—Ç—Ä–µ–±—É—î –∂–æ–¥–Ω–∏—Ö –æ–±–ª—ñ–∫–æ–≤–∏—Ö –¥–∞–Ω–∏—Ö –∞–±–æ —Ç–æ–∫–µ–Ω—ñ–≤ –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó –¥–ª—è –¥–æ—Å—Ç—É–ø—É –¥–æ –ø–µ–≤–Ω–æ–≥–æ —Ä–µ—Å—É—Ä—Å—É –∞–±–æ —Å–µ—Ä–≤—ñ—Å—É. –¶–µ –æ–∑–Ω–∞—á–∞—î, —â–æ –±—É–¥—å-—Ö—Ç–æ –º–æ–∂–µ –æ—Ç—Ä–∏–º–∞—Ç–∏ –¥–æ—Å—Ç—É–ø –¥–æ —Ä–µ—Å—É—Ä—Å—É –∞–±–æ —Å–µ—Ä–≤—ñ—Å—É –±–µ–∑ –±—É–¥—å-—è–∫–∏—Ö –æ–±–º–µ–∂–µ–Ω—å –∞–±–æ –ø–µ—Ä–µ–≤—ñ—Ä–∫–∏ –π–æ–≥–æ —ñ–¥–µ–Ω—Ç–∏—á–Ω–æ—Å—Ç—ñ.
An authentication method that does not require any credentials or authentication tokens to access a specific resource or service. This means that anyone can access the resource without restrictions or identity verification.

`AUTH_TOKEN` ::
//–°–∫–æ—Ä–æ—á–µ–Ω–Ω—è –≤—ñ–¥ ¬´—Ç–æ–∫–µ–Ω –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó¬ª -- —Ü–µ —Ñ—Ä–∞–≥–º–µ–Ω—Ç –¥–∞–Ω–∏—Ö, —è–∫–∏–π –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î—Ç—å—Å—è –¥–ª—è –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ –∞–±–æ —Å–∏—Å—Ç–µ–º–∏ –¥–ª—è –¥–æ—Å—Ç—É–ø—É –¥–æ –ø–µ–≤–Ω–æ–≥–æ —Å–µ—Ä–≤—ñ—Å—É —á–∏ —Ä–µ—Å—É—Ä—Å—É. –¢–æ–∫–µ–Ω–∏ –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó –∑–∞–∑–≤–∏—á–∞–π –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—é—Ç—å —É –≤–µ–±–¥–æ–¥–∞—Ç–∫–∞—Ö, API —Ç–∞ —ñ–Ω—à–∏—Ö –º–µ—Ä–µ–∂–µ–≤–∏—Ö —Å–∏—Å—Ç–µ–º–∞—Ö –¥–ª—è –∑–∞–±–µ–∑–ø–µ—á–µ–Ω–Ω—è –±–µ–∑–ø–µ—á–Ω–æ–≥–æ —Ç–∞ –µ—Ñ–µ–∫—Ç–∏–≤–Ω–æ–≥–æ –¥–æ—Å—Ç—É–ø—É –¥–æ —Ä–µ—Å—É—Ä—Å—ñ–≤. –¢–æ–∫–µ–Ω–∏ –º–æ–∂—É—Ç—å –ø—Ä–∏–π–º–∞—Ç–∏ —Ä—ñ–∑–Ω—ñ —Ñ–æ—Ä–º–∏, –Ω–∞–ø—Ä–∏–∫–ª–∞–¥ –≤–∏–ø–∞–¥–∫–æ–≤—ñ —Ä—è–¥–∫–∏ —Å–∏–º–≤–æ–ª—ñ–≤, –∑–∞—à–∏—Ñ—Ä–æ–≤–∞–Ω—ñ –¥–∞–Ω—ñ –∞–±–æ –Ω–∞–≤—ñ—Ç—å –≤–µ–±—Ç–æ–∫–µ–Ω–∏ JSON (JWT), —è–∫—ñ –º—ñ—Å—Ç—è—Ç—å —ñ–Ω—Ñ–æ—Ä–º–∞—Ü—ñ—é –ø—Ä–æ –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ —Ç–∞ —Ç–µ—Ä–º—ñ–Ω –¥—ñ—ó.
AUTH_TOKEN is short for "authentication token," which is a fragment of data used to grant a user or system access to a specific resource or service. Authentication tokens are typically used in web applications, APIs, and other network systems to ensure secure and efficient access to resources. Tokens come in various forms, such as random strings of characters, encrypted data, or JSON web tokens (JWT) that contain information about the user and expiration time.

`BEARER` ::
//Bearer-–∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—è —î –º–µ—Ç–æ–¥–æ–º –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —è–∫–∏–π –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î—Ç—å—Å—è –≤ –∫–æ–º—É–Ω—ñ–∫–∞—Ü—ñ—ó –Ω–∞ –æ—Å–Ω–æ–≤—ñ `HTTP`. –í–æ–Ω–∞ –ø–æ–ª—è–≥–∞—î –≤ —Ç–æ–º—É, —â–æ –∑–∞–ø–∏—Ç –º—ñ—Å—Ç–∏—Ç—å —Ç–æ–∫–µ–Ω –±–µ–∑–ø–µ–∫–∏, –≤—ñ–¥–æ–º–∏–π —è–∫ ¬´Bearer-—Ç–æ–∫–µ–Ω¬ª, —É –∑–∞–≥–æ–ª–æ–≤–∫—É –¥–ª—è –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó –∑–∞–ø–∏—Ç–∞.
BEARER authentication method is used in HTTP-based communication. It is based on security tokens called bearer tokens.
+
//–£ —Ü—å–æ–º—É –º–µ—Ç–æ–¥—ñ –∫–ª—ñ—î–Ω—Ç, —è–∫–∏–π –∑–∞–ø–∏—Ç—É—î –¥–æ—Å—Ç—É–ø –¥–æ –∑–∞—Ö–∏—â–µ–Ω–æ–≥–æ —Ä–µ—Å—É—Ä—Å—É, –≤–∫–ª—é—á–∞—î —Ç–æ–∫–µ–Ω –¥–æ—Å—Ç—É–ø—É –≤ –∑–∞–≥–æ–ª–æ–≤–æ–∫ `Authorization` –∑–∞–ø–∏—Ç–∞, —è–∫–∏–π –∑–∞–∑–≤–∏—á–∞–π —î –¥–æ–≤–≥–∏–º —Ä—è–¥–∫–æ–º —Å–∏–º–≤–æ–ª—ñ–≤, —â–æ –ø—Ä–µ–¥—Å—Ç–∞–≤–ª—è—î —ñ–¥–µ–Ω—Ç–∏—á–Ω—ñ—Å—Ç—å —Ç–∞ –¥–æ–∑–≤–æ–ª–∏ –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞. –°–µ—Ä–≤–µ—Ä –ø–æ—Ç—ñ–º –ø–µ—Ä–µ–≤—ñ—Ä—è—î –ø—Ä–∞–≤–∏–ª—å–Ω—ñ—Å—Ç—å —Ç–æ–∫–µ–Ω–∞, —ñ —è–∫—â–æ –≤—ñ–Ω —î –¥—ñ–π—Å–Ω–∏–º, –Ω–∞–¥–∞—î –¥–æ—Å—Ç—É–ø –¥–æ –∑–∞—Ö–∏—â–µ–Ω–æ–≥–æ —Ä–µ—Å—É—Ä—Å—É.
In this method, the client requesting access to a secured resource includes a bearer token in the request's `Authorization` header, which is usually a long string of characters representing the user's identity and permissions. The server then verifies the token's validity and grants access to the secured resource.
+
//Bearer-—Ç–æ–∫–µ–Ω –º–æ–∂–µ –±—É—Ç–∏ –æ—Ç—Ä–∏–º–∞–Ω–∏–π —á–µ—Ä–µ–∑ –æ–∫—Ä–µ–º–∏–π –ø—Ä–æ—Ü–µ—Å –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —Ç–∞–∫–∏–π —è–∫ OAuth –∞–±–æ OpenID Connect, —ñ –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î—Ç—å—Å—è –≤ API –¥–ª—è —É–ø—Ä–∞–≤–ª—ñ–Ω–Ω—è –¥–æ—Å—Ç—É–ø–æ–º –¥–æ —Ä–µ—Å—É—Ä—Å—ñ–≤. –û—Å–∫—ñ–ª—å–∫–∏ Bearer-—Ç–æ–∫–µ–Ω –≤–∫–ª—é—á–∞—î—Ç—å—Å—è –≤ –∑–∞–≥–æ–ª–æ–≤–æ–∫ –∑–∞–ø–∏—Ç–∞, —Ç–æ –º–æ–∂–µ –±—É—Ç–∏ –ª–µ–≥–∫–æ –±—É—Ç–∏ –ø–µ—Ä–µ—Ö–æ–ø–ª–µ–Ω–∏–π. –¢–æ–º—É –≤–∞–∂–ª–∏–≤–æ –∑–∞–±–µ–∑–ø–µ—á–∏—Ç–∏ –±–µ–∑–ø–µ—á–Ω—É –π–æ–≥–æ –ø–µ—Ä–µ–¥–∞—á—É, –Ω–∞–ø—Ä–∏–∫–ª–∞–¥, –∑–∞ –¥–æ–ø–æ–º–æ–≥–æ—é HTTPS-–∑'—î–¥–Ω–∞–Ω–Ω—è.
The bearer token can be obtained through a separate authentication process, such as OAuth or OpenID Connect, and is used in APIs for managing access to resources. Since the bearer token is included in the request header, it can be easily intercepted. Therefore, it is essential to ensure its secure transmission, for example, by using an HTTPS connection.

`BASIC` ::
//BASIC-–∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—è -- —Ü–µ –º–µ—Ç–æ–¥ –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —è–∫–∏–π –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î—Ç—å—Å—è –≤ `HTTP`-–ø—Ä–æ—Ç–æ–∫–æ–ª—ñ –¥–ª—è –∑–∞—Ö–∏—Å—Ç—É –≤–µ–±—Ä–µ—Å—É—Ä—Å—ñ–≤ –≤—ñ–¥ –Ω–µ—Å–∞–Ω–∫—Ü—ñ–æ–Ω–æ–≤–∞–Ω–æ–≥–æ –¥–æ—Å—Ç—É–ø—É. –¶–µ–π –º–µ—Ç–æ–¥ –ø–µ—Ä–µ–¥–±–∞—á–∞—î –ø–µ—Ä–µ–¥–∞—á—É —ñ–º–µ–Ω—ñ –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ —Ç–∞ –ø–∞—Ä–æ–ª—è —É –≤–∏–≥–ª—è–¥—ñ –Ω–µ–∑–∞—à–∏—Ñ—Ä–æ–≤–∞–Ω–æ–≥–æ —Ç–µ–∫—Å—Ç—É —É –∑–∞–≥–æ–ª–æ–≤–∫—É HTTP-–∑–∞–ø–∏—Ç–∞.
BASIC authentication method is used in the HTTP protocol to protect web resources from unauthorized access. This method involves transmitting a username and password as unencrypted text in the HTTP request header.
+
//–£ BASIC-–∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —Å–µ—Ä–≤–µ—Ä –∑–∞–ø–∏—Ç—É—î —ñ–º'—è –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ —Ç–∞ –ø–∞—Ä–æ–ª—å –≤—ñ–¥ –∫–ª—ñ—î–Ω—Ç–∞, –ø—ñ—Å–ª—è —á–æ–≥–æ –ø–µ—Ä–µ–≤—ñ—Ä—è—î —ó—Ö –≤—ñ–¥–ø–æ–≤—ñ–¥–Ω—ñ—Å—Ç—å –∑–±–µ—Ä–µ–∂–µ–Ω–∏–º –≤—ñ–¥–æ–º–æ—Å—Ç—è–º –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó. –Ø–∫—â–æ —ñ–º'—è –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ —Ç–∞ –ø–∞—Ä–æ–ª—å –∑–±—ñ–≥–∞—é—Ç—å—Å—è –∑ –≤—ñ–¥–æ–º–æ—Å—Ç—è–º–∏ –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —Å–µ—Ä–≤–µ—Ä –Ω–∞–¥–∞—î –¥–æ—Å—Ç—É–ø –¥–æ –∑–∞—Ö–∏—â–µ–Ω–æ–≥–æ —Ä–µ—Å—É—Ä—Å—É. –Ø–∫—â–æ –∂ –¥–∞–Ω—ñ –Ω–µ –∑–±—ñ–≥–∞—é—Ç—å—Å—è, —Å–µ—Ä–≤–µ—Ä –ø–æ–≤–µ—Ä—Ç–∞—î –∫–æ–¥ –ø–æ–º–∏–ª–∫–∏ (–∑–∞–∑–≤–∏—á–∞–π —Ü–µ `401`).
In BASIC authentication, the server requests the username and password from the client, and then verifies that they match the authentication records. If the username and password match the authentication records, the server grants access to the protected resource. If records do not match, the server returns an error code (usually `401`).
+
//–ë—ñ–ª—å—à—ñ—Å—Ç—å –≤–µ–±—Å–µ—Ä–≤–µ—Ä—ñ–≤ —Ç–∞ –¥–æ–¥–∞—Ç–∫—ñ–≤ –ø—ñ–¥—Ç—Ä–∏–º—É—é—Ç—å BASIC-–∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—é. –û–¥–Ω–∞–∫, –æ—Å–∫—ñ–ª—å–∫–∏ —ñ–º'—è –∫–æ—Ä–∏—Å—Ç—É–≤–∞—á–∞ —Ç–∞ –ø–∞—Ä–æ–ª—å –ø–µ—Ä–µ–¥–∞—é—Ç—å—Å—è —É –≤—ñ–¥–∫—Ä–∏—Ç–æ–º—É –≤–∏–≥–ª—è–¥—ñ, BASIC-–∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—è –Ω–µ —î –Ω–∞–¥—ñ–π–Ω–∏–º –º–µ—Ç–æ–¥–æ–º –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó. –¢–æ–º—É –≤–∞–∂–ª–∏–≤–æ –∑–∞–±–µ–∑–ø–µ—á–∏—Ç–∏ –∑–∞—Ö–∏—Å—Ç –∑'—î–¥–Ω–∞–Ω–Ω—è —à–∏—Ñ—Ä—É–≤–∞–Ω–Ω—è–º, –Ω–∞–ø—Ä–∏–∫–ª–∞–¥, –∑–∞ –¥–æ–ø–æ–º–æ–≥–æ—é –ø—Ä–æ—Ç–æ–∫–æ–ª—É HTTPS.
Most web servers and applications support BASIC authentication. However, since the username and password are sent in plain text, BASIC authentication is not a reliable authentication method. Therefore, it is essential to ensure connection protection through encryption, for example, using an HTTPS protocol.

`AUTH_TOKEN+BEARER` ::
//–ö–æ–º–±—ñ–Ω–æ–≤–∞–Ω–∏–π –º–µ—Ç–æ–¥ –∞—É—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó, —â–æ –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î—Ç—å—Å—è –¥–ª—è –∑–∞—Ö–∏—Å—Ç—É –¥–æ—Å—Ç—É–ø—É –¥–æ API. –¶–µ–π –º–µ—Ç–æ–¥ –≤–∏–∫–æ—Ä–∏—Å—Ç–æ–≤—É—î –¥–≤–∞ —Ç–∏–ø–∏ —Ç–æ–∫–µ–Ω—ñ–≤: "—Ç–æ–∫–µ–Ω –¥–æ—Å—Ç—É–ø—É" (BEARER token) —Ç–∞ "—Ç–æ–∫–µ–Ω –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó" (`AUTH_TOKEN`). –í–∏–∫–æ—Ä–∏—Å—Ç–∞–Ω–Ω—è —Ü—å–æ–≥–æ –º–µ—Ç–æ–¥—É –ø–µ—Ä–µ–¥–±–∞—á–∞—î –¥–≤–æ–µ—Ç–∞–ø–Ω—É –∞–≤—Ç–æ—Ä–∏–∑–∞—Ü—ñ—é –∑ –æ—Ç—Ä–∏–º–∞–Ω–Ω—è–º —Ç–æ–∫–µ–Ω–∞ –¥–æ—Å—Ç—É–ø—É (`BEARER`) –Ω–∞ –æ—Å–Ω–æ–≤—ñ —Ç–æ–∫–µ–Ω–∞ –∞–≤—Ç–µ–Ω—Ç–∏—Ñ—ñ–∫–∞—Ü—ñ—ó (`AUTH_TOKEN`).
A combined authentication method is usually used to protect access to APIs. This method uses two types of tokens: the "access token" (bearer token) and the "authentication token" (`AUTH_TOKEN`). This method involves a two-step authorization process with obtaining an access token (`BEARER`) based on an authentication token (`AUTH_TOKEN`).

//–ß–∞—Å "–∂–∏—Ç—Ç—è" —Ç–æ–∫–µ–Ω–∞ –≤–∏–∑–Ω–∞—á–∞—î—Ç—å—Å—è –∑–∞ –¥–æ–ø–æ–º–æ–≥–æ—é JWT-–∫–ª–µ–π–º–∞ *`exp`* (expire time), —è–∫–∏–π –º—ñ—Å—Ç–∏—Ç—å—Å—è –≤ –∞–≤—Ç–æ—Ä–∏–∑–∞—Ü—ñ–π–Ω–æ–º—É —Ç–æ–∫–µ–Ω—ñ. –¶–µ –≤—ñ–¥–ø–æ–≤—ñ–¥–∞—î —Å–ø–µ—Ü–∏—Ñ—ñ–∫–∞—Ü—ñ—ó JWT, –≤–∏–∑–Ω–∞—á–µ–Ω—ñ–π —É https://www.rfc-editor.org/rfc/rfc7519[RFC 7519].

If the `exp` claim is absent, caching is not performed.
